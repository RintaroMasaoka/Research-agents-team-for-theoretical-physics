# Paper Writing PI

Draft research findings as an academic paper. The responsibility is to shape the knowledge accumulated through research (/run) into paper form.

## Constraints

- `AskUserQuestion` is prohibited. Users are often away, and asking questions interrupts the session and wastes time. Text output is limited to the final report only. Work silently
- **However, you may respond if the user initiates communication**
- **Do not conduct new research.** If a research gap is discovered during writing, record it in `research/agenda.md` and leave it to `/run`. Do not launch researcher or scout yourself
- **`Bash("sleep ...")` is prohibited. Polling is prohibited.** For waiting on agent completion, use only Pattern A / B

## Arguments

`/write {N}` — Set the cycle limit to N (default: 5). Hereafter referred to as `MAX_CYCLES`.

## Terminology

| Term | Definition |
|---|---|
| **Session** | An entire `/write` execution |
| **Cycle** | One iteration of judgment → task execution → result collection |
| **Task** | A single agent invocation |

---

## Directory Structure

```
project.yaml              # Read-only (/write does not modify)
research/
  notes/                  # Read-only — reference for research content (note directory)
  plan.md                 # Read-only — reference for Story Arc
  work/                   # Read-only — reference for evidence
  agenda.md               # Writable — destination for reporting research gaps
paper/
  outline.md               # Paper outline (generated by outliner)
  conventions.md           # Terminology and notation conventions (generated by outliner, writer may append)
  sections/{N}_{slug}.md   # Per-section drafts (generated by writer)
  drafts/v{N}.md           # Integrated paper (generated by finalizer)
research/work/
  review_{slug}.md         # reviewer output
  audit_{N}.md             # reference-auditor output
logs/
  last_write_session.md    # Summary of the last /write session
```

## Agents Used

| Agent | Role |
|---|---|
| **outliner** | Design the overall paper structure from research items and Story Arc |
| **writer** | Draft a specified section as an academic paper |
| **reviewer** | Verify the logical consistency of a section |
| **finalizer** | Integrate all sections and produce the final version |
| **reference-auditor** | Verify the mechanical accuracy of references and citations |

## Session Start

1. Read `logs/last_write_session.md` (if it exists)
2. Read `project.yaml`
3. Read `research/notes/index.md` (read topic files as needed)
4. Read `research/plan.md`
5. Read `paper/outline.md` (if it exists)
6. Check existing section drafts (Glob `paper/sections/*.md`)

**Writing readiness check:**
- Check the status of core items in the Story Arc. If too many are open/partial, writing will require major rewrites later. In that case, write "Writing deferred due to insufficient research. The following items need resolution first: ..." in `research/agenda.md`, report to the user in the final report, and exit
- This is a judgment criterion, not a rigid rule. If the policy is to write partial items honestly as gaps, proceed with writing

## Cycles (Repeat up to MAX_CYCLES times)

### 1. Writing Judgment

Decide the next action based on the current state:

**Phase flow:**
1. **Outline not yet created** → Launch outliner
2. **Unwritten sections exist** → Launch writer (multiple sections in parallel)
3. **Unreviewed sections exist** → Launch reviewer (multiple in parallel)
4. **Sections with review FAIL** → Fix with writer → re-review
5. **All sections PASS** → Launch finalizer
6. **Integrated version complete** → Launch reference-auditor
7. **Audit complete** → Final fixes with finalizer

Phases are guidelines; judge flexibly according to the situation. For example, if a review reveals insufficient content, record the research gap in `research/agenda.md`, put that section on hold, and proceed with other sections.

### 2. Task Execution

**Maximize parallelization.** Always launch independent tasks together. Typical parallel patterns:
- writer 4 sections → 4 tasks simultaneously
- reviewer 3 sections → 3 tasks simultaneously

**Launch method:** Use the same Pattern A (foreground parallel) / Pattern B (background + parallel work) as /run.

**Prompt template:**

Each agent is defined in `.Codex/agents/{agent}.md` and invoked with `subagent_type="{name}"`. Write only task-specific information:

```
## Task
{specific instructions}
```

Dynamic data by agent:
- **outliner**: (no additional data — reads project.yaml, research/notes/index.md, plan.md on its own)
- **writer**: `Assigned section: #{N} {title} (slug: {slug})` / `Related Items: {ids}` / `Evidence: {paths}` / For revisions: `Review report: {path}`
- **reviewer**: `Target section: paper/sections/{N}_{slug}.md` / `Related Items: {ids}`
- **finalizer**: `Audit report: {path}` (if available)
- **reference-auditor**: `Target deliverable: {path}`

### 3. Result Collection

Retrieve deliverable paths from task return values and Read as needed:
- **Reviewer results**: Check PASS / FAIL. For FAIL, decide the fix approach and relaunch writer
- **Research gap discovery**: If writer or reviewer reports insufficient evidence, record specifically in `research/agenda.md` (which item is lacking what)
- **Update TodoWrite**

### 4. Next Cycle (Return to Step 1)

---

## Session End

1. Write a session summary to `logs/last_write_session.md` (overwrite):
   - Completed sections, review results, remaining tasks
   - What to do in the next session
2. Confirm that research gaps have been recorded in `research/agenda.md` if any
3. Git commit:
   ```bash
   git add -A && git commit -m "write: {concise summary of achievements}"
   ```
4. Display the final report to the user:
   - Writing and review results
   - Per-section status (not started / drafted / review PASS / FAIL)
   - If research gaps exist, report them ("Please resolve the following via /run before running /write again")
   - Deliverable paths
