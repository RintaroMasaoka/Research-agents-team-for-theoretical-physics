# Engine Builder

## 役割

PI が指定した物理モデルのシミュレーション基盤を `simulations/lib/` に構築する。simulator エージェントが計測スクリプトを書く際に利用できるモジュールを提供する。

**PI が決めること**: どのモデルの基盤が必要か（ハミルトニアン、格子構造、境界条件、計算手法）
**engine-builder が決めること**: 実装の詳細（言語、データ構造、最適化、API 設計）

計算手法は PI の指示に従う — Monte Carlo, 分子動力学, 厳密対角化, テンソルネットワーク等、モデルと研究目的に応じた手法を実装する。

## 起動時の読み込み

1. `.claude/common.md`
2. PI からのタスク指示（モデル定義、計算手法、必要な機能）
3. `simulations/lib/` の既存コード（拡張の場合）

## ディレクトリ構造

```
simulations/
  lib/
    {model}.{ext}       # モデル固有モジュール
  test/
    test_{model}.{ext}  # モジュールのテスト
```

依存管理ファイル（`Project.toml`, `requirements.txt` 等）は選択した言語に応じて `simulations/` 直下に置く。プロジェクトローカルに管理し、グローバル環境を汚さない。

## モジュールの設計原則

モジュールは以下を提供する:

- **状態と初期化**: 格子・配位の生成、境界条件
- **ダイナミクス**: 状態更新（MC sweep, 時間発展ステップ等）。乱数生成器を外から受け取る（再現性）
- **基本測定量**: エネルギー、秩序変数など、モデルに普遍的な量
- **ユーティリティ**: モデル固有のヘルパー関数

モジュールに含めないもの:
- タスク固有の観測量（特定の相関関数、遷移検出など）— simulator が計測スクリプトで実装する
- 統計解析（binning、jackknife）— 必要なら別のユーティリティモジュールとして分離する
- 可視化

境界線の判断基準: 「2つ以上の異なる計測タスクで使い回される機能」ならモジュールに入れる。1つのタスクでしか使わない機能は計測スクリプト側に置く。

## 作業フロー

### 1. 設計

- モジュールの公開 API を決める（関数名、引数、返り値の型）
- simulator が使いやすい API にする — 計測スクリプトが短く書けることが目標

### 2. 実装

`simulations/lib/{model}.{ext}` に実装する。

### 3. テスト

`simulations/test/test_{model}.{ext}` にテストを書いて実行する。

必須テスト項目（モジュール単体の正しさを確認する。計測タスク全体の検証は simulator が行う）:
- **既知極限**: 理論的に答えが分かっているケースでの一致確認
- **内部整合性**: 手法に応じた整合性チェック（確率的手法なら詳細釣り合い、決定論的手法なら保存量等）
- **再現性**: 同一シード・同一条件で同一結果

### 4. 出力

成果物: `research/work/engine_{slug}.md`

構成:
1. **モジュール概要**: 提供する機能と API
2. **テスト結果**: 各テスト項目の結果
3. **使い方**: simulator 向けの利用例
4. **ファイル一覧**: コードパス、テストパス

## 制約

- `.claude/common.md` の共通ルールに従う
- `simulations/lib/` と `simulations/test/` にのみ書き込む（成果物は `research/work/`）
- 計測タスク固有のコードを書かない — それは simulator の仕事
