# ワーカー共通ルール

## 研究アイテム（project.yaml）

`project.yaml` の `items:` にはネスト構造のアイテムが管理されている。各アイテムには `kind`（narrative, task, question, conjecture, example, caution, gap, observation 等）と `status`（open, active, partial, resolved, reframed, dropped）がある。

`detail` フィールドはステータスの補足説明であり、構造化された `status` では伝えきれないニュアンスを自由記述する。特に `resolved` では厳密性と確信度を記述する — 後のセッションで PI がその結果にどこまで依拠してよいかを判断する材料になる。`open` や `dropped` でも状況の説明があれば書く。

アイテムの status 判定は研究全体の文脈が必要な判断であり、PI の責任で行う。

- **読み取り専用**（PI のみが書き込む）
- アイテムの状態変更を提案する場合は、成果物ファイルに根拠とともに記述する
- 正直な報告が最重要: 解決できていないものを resolved と提案してはならない

## エージェント間通信

プロンプトに大量のデータを載せるとコンテキストウィンドウを浪費する。ファイルに書き出せば必要な部分だけ Read できる。

- **入力**: PI からタスク指示とファイルパスを受け取り、必要なデータは自分で Read する
- **出力**: 成果物をファイルに書き出し、Task の返り値として `DONE: {成果物パス}` を返す
- 生データをプロンプトに直接載せない — 常にファイル経由で通信する

## ログ（人間用の進捗記録）

`logs/{agent}_{timestamp}.md` に作業の要約を Write する。
これは人間が後から進捗を振り返るための記録であり、PI の判断材料ではない。

## 制約

- 数式は LaTeX 記法で書く（インラインは `$...$`、ディスプレイは `$$...$$`）。prose 中に生の変数名や式を地の文として埋め込まず、常に `$...$` で囲む。Markdown ファイル上での可読性を優先し、LaTeX 環境名ではなく `$$` を使う
- AskUserQuestion の使用禁止（ユーザーは `/run`・`/write` 中に席を外していることが多く、質問するとセッションが途切れる。ただしユーザーから話しかけられた場合は応答してよい）
- プロジェクトディレクトリ外への書き込み禁止
- 各エージェントは自分の出力先にのみ書き込む（各エージェント定義を参照）
