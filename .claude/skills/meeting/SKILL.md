# ミーティング

引数: $ARGUMENTS

---

## フロー

**原則: 質問を先に出し、データ読み込みは回答後に必要な分だけ行う。** ミーティングはリアルタイムの対話であり、応答の遅延はユーザーの思考を途切れさせる。質問には AskUserQuestion を使う（テキスト出力での質問は応答の取りこぼしが起きるため使わない）。

```
初期化
    ▼ データ読み込み: project.yaml
    ▼ 状況に応じた開始
        ├─ テーマ未設定 → テーマ設定から始める
        └─ テーマ設定済み → 進捗レポートを提示して議論を開く
    ▼ ユーザーと自由に議論
    ▼ 決定事項を都度反映
```

### 初期化

セッション開始時に以下を実行する（→ 逐次記録の原則）。

1. `Bash("date '+%Y-%m-%dT%H:%M'")` で現在日時を取得する（以降、この値をセッション内の全タイムスタンプに使う）
2. `project.yaml` の `last_meeting` を取得した日時に更新
3. `meetings/YYYY-MM-DD_HHMM.md` を取得した日時で作成（ヘッダのみ）:

```markdown
# ミーティング YYYY-MM-DD HH:MM

## 議論事項

## 決定事項

## 反映済み変更
```

初期化が終わったら `meeting: init YYYY-MM-DD HH:MM` でコミットする。

### テーマ未設定の場合

研究テーマと方向性を設定する。

```
AskUserQuestion: 研究テーマの概要を Other に記述してもらう
    ▼ AI がアプローチを数案提示 → 方向性を掘り下げ（2-3ラウンド）
    ▼ project.yaml + plan.md に反映
```

**反映先:**
- `project.yaml` に topic, field, created_at, questions を設定
- `research/plan.md` に研究計画を作成
- Git 未初期化なら `git init && git add -A && git commit -m "init: {topic}"`

**原則:** ユーザーが決めるのは「何を」「なぜ」。「どうやって」は AI が決める。questions の文言は AI が起案しユーザーに確認。

### テーマ設定済みの場合

研究全体の方向性を確認し、議論を開く。

```
データ読み込み: research/notes/index.md + plan.md + 最新議事録
    ▼ agenda.md（PI が /run 中に蓄積した議題）があれば読み込み → 即削除（消費済みの議題が次回に残らないようにする）
    ▼ 進捗レポートを提示
    ▼ 読み込んだ議題があれば併せて表示・議論
    ▼ AI が大局的な観察を示す
    ▼ ユーザーと自由に議論
    ▼ 決定事項を反映
```

**進捗レポート:**
```
テーマ: {topic}
Research Questions:
  resolved: {N}件 — {主要な findings}
  partial: {N}件 — {主要な gaps}
  open: {N}件
前回以降の主な成果: [要約]
```

**反映先:**
- 構造的な変更（ストーリー・ステップの追加/削除/修正、Thesis の書き直し）→ `research/plan.md` を直接編集 + `project.yaml` を更新。変更箇所には `> [Meeting YYYY-MM-DD] {変更理由}` マーカーを残し、PIが変更の文脈を理解できるようにする
- 姿勢・アプローチ・トーンの指示（plan.md に書きにくいもの）→ `research/feedback.md`

**原則:** 個別 question の管理ではなく全体の方向性。「この研究で何を言いたいか」の擦り合わせ。

## 対話の深め方

AI が報告して終わりではなく、ユーザーの視点を引き出して方向性を共に形作る。以下の場面では能動的にユーザーの判断を求める。

**方向性の転換点:** plan.md の構造変更（ステップ・thesis・question の追加/削除）は反映前にユーザーの承認を得る。変更の内容・理由・現状維持の選択肢を示す。

**曖昧な発言:** ユーザーの発言が多義的なとき、AI の解釈を明示して確認を取る。抽象的な合意で止めず、plan.md のどこがどう変わるかまで具体化してから反映する。

**成果の解釈:** 進捗レポート後、AI の評価を述べるだけでなく、ユーザーにその成果のストーリーライン上の位置づけを問う。評価のずれがあればそれ自体を議論の材料にし、plan.md の構造や重点配分への影響まで掘り下げる。

---

## 逐次記録の原則

ユーザーは区切りのよいところでそのまま離脱することがある。最後にまとめて書く後処理は実行されないリスクが高い。**記録と反映はその場で行い、コミットまで済ませる。**

| タイミング | やること |
|---|---|
| セッション開始時 | 議事録ファイル作成 + `last_meeting` 更新（→ 初期化セクション参照） |
| 議題が出たとき | 議事録の「議論事項」に Edit で追記 |
| 決定が出たとき | 議事録の「決定事項」に追記 + 該当ファイルに即座に反映 |
| ファイルを変更したとき | 議事録の「反映済み変更」に追記 + git commit |

**Git コミット:** 変更したファイルを `git add` で個別指定し（意図しないファイルの混入防止）、`meeting: {変更内容の要約}` の形式でコミットする。git log でミーティング起因の変更を追跡可能にするための接頭辞。
