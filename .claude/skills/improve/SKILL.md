# システム改善

エージェントやスキルの出力品質を改善する。ユーザーは「何が不満か」を伝えるだけ。AI が修正を設計・実行し、独立したエージェントが品質を検証する。

引数: $ARGUMENTS

---

## フロー

```
引数あり → 不満点として扱う → ニュアンス確認 → 修正の実行
引数なし → AskUserQuestion で意図を確認（下記3択 + Other で不満点を自由記述）

1.「全体をレビューしてほしい」:
    ▼ データ読み込み: 全エージェント定義 + SKILL.md
    ▼ AI がレビューし改善点を提示
    ▼ AskUserQuestion: 採用する改善を選択
    ▼ 修正の実行

2.「特定の対象をレビューしてほしい」:
    ▼ AskUserQuestion: 対象を指定してもらう（エージェント名・スキル名・ファイルパス等）
    ▼ データ読み込み: 該当ファイルのみ
    ▼ AI がレビューし改善点を提示
    ▼ AskUserQuestion: 採用する改善を選択
    ▼ 修正の実行

3. 不満点がある（Other に記述）:
    ▼ データ読み込み: 不満に関連するファイルのみ
    ▼ ユーザーとニュアンスを擦り合わせてから修正方針を決める
    ▼ 修正の実行
```

**反映先:** エージェント → `.claude/agents/{agent}.md` / ワークフロー → `.claude/skills/{skill}/SKILL.md`

---

## 修正の実行

### 1. 書き直し

修正対象ファイルを Write で全体書き直す。プロンプトファイルは小さく（通常 < 200行）、改善時は構造変更を伴うことが多い。全体を一度に生成することで、ファイル全体の一貫性を保ちやすい。

書き直す際の心得（アーキテクチャ原則はプロンプト設計仕様を参照）:
- 正しいやり方を自然に記述する。「X をするな」という禁止形ではなく、あるべき姿を書く
- 修正点の重要度に比例した分量で書く。1行で済む修正に1段落使わない
- 指示には理由を添える。理由があれば AI はエッジケースでも判断できる。理由がなければ教義として硬直化する

### 2. 検証エージェント

書き直したファイルを `subagent_type: "reviewer"` の Agent で検証させる。reviewer は対象を読み取り専用で検証し指摘のみ行うエージェントで、修正者とは独立した視点を提供する。修正を行った本人は問題の文脈を知っているため、一貫性の崩れに気づきにくい。文脈を持たないエージェントに「初見読者としてファイル全体の一貫性を検証」させる。

プロンプト:
```
プロンプトファイルを初見の読者として読み、以下の観点で問題を指摘してください。

対象ファイル: {path}

特に注目してほしい箇所: {今回の修正箇所}

## 検証観点

1. **痕跡**: 過去のインシデントへの言及・具体例が残っていないか。
   初見の読者に「なぜこの例？」と違和感を与える記述
2. **バランス**: 特定の箇所がその重要性に比べて不釣り合いに強調されていないか。
   後から追加された感のある段落
3. **理由なき指示**: 「こうしろ」だけで「なぜ」が書かれていない指示
4. **重複・分散**: 同じ趣旨の記述が複数箇所に分散していないか

各指摘には該当箇所の引用と改善案を含めてください。
問題がなければ「問題なし」と報告してください。
```

### 3. 検証結果の反映

検証エージェントの指摘をユーザーに報告し、妥当な指摘があれば修正する。

### 4. ユーザー確認

最終結果を AskUserQuestion で承認を得てからコミットする。

## プロンプト設計仕様

プロンプトファイルのアーキテクチャ原則。書き直し時にこれらを満たしているか確認する。

1. 役割の分離 — 各ファイルは自分の責務だけ。共通ルールは `.claude/common.md` に一度だけ
2. 低結合 — ファイル間の依存を最小限に
3. ファイルパス通信 — 生データをプロンプトに載せず、ファイルに書き出してパスを渡す（プロンプトの肥大化を防ぎ、データの再利用を可能にする）

## Git コミット

修正を反映したら git commit する。
- 変更したファイルを `git add` で個別指定する（意図しないファイルの混入防止）
- メッセージ形式: `improve: {変更内容の要約}`
